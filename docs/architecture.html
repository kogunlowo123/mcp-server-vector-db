Executive Summary

mcp-server-vector-db is a Model Context Protocol server that provides a unified interface to four vector database providers — Pinecone, Weaviate, Qdrant, and ChromaDB — through six MCP tools. The architecture uses a provider abstraction pattern: a VectorProvider interface defines operations (upsert, query, delete, list/create/stats collections) and four concrete implementations adapt each database's native SDK. The active provider is selected at runtime via the VECTOR_PROVIDER environment variable, with lazy initialization on first tool call. This enables AI assistants to perform semantic search, manage embeddings, and administer vector collections without provider-specific knowledge.

Data / Request Flow

  1. AI assistant sends MCP tool-call request over stdin
  2. StdioServerTransport deserializes JSON-RPC message
  3. McpServer routes to matching tool handler
  4. Tool handler calls getProvider() — lazy singleton that reads VECTOR_PROVIDER, VECTOR_API_KEY, VECTOR_HOST, VECTOR_URL from environment
  5. createProvider() factory instantiates the correct provider class (validates required config)
  6. Provider method is called (e.g., provider.query(collection, vector, topK, filter))
  7. Provider translates to native SDK call (Pinecone Index.query, Weaviate GraphQL, Qdrant search, ChromaDB collection.query)
  8. Results are normalized to standard types (QueryResult, CollectionInfo, etc.) and returned as JSON

Security Controls

  • Input validation: Zod schemas on all tools; vectors array validated with nested object schemas
  • API key management: Provider credentials read from environment variables, never hardcoded
  • Provider isolation: Each provider validates required config (apiKey, host, url) at creation time
  • Lazy initialization: Provider created only on first tool call, not at server startup
  • Error boundaries: Every tool has try/catch returning isError:true with descriptive messages
  • Minimal container: node:20-alpine with only production dependencies
  • Score normalization: ChromaDB converts distance to similarity (1 - distance) to match other providers

Deployment

  • Docker: docker build -t mcp-server-vector-db . — multi-stage alpine build, defaults to ChromaDB at localhost:8000
  • Standalone: npm ci && npm run build && VECTOR_PROVIDER=pinecone VECTOR_API_KEY=xxx node dist/index.js
  • Dev mode: npm run dev (ts-node)
  • With ChromaDB: Run ChromaDB container alongside: docker run -p 8000:8000 chromadb/chroma
  • MCP client: Configure stdio transport with appropriate VECTOR_* environment variables
